# WYB_BISHE 项目说明

## 1. 项目概述

本项目面向多站点、多气象要素的时空预测任务，核心研究主题为：

- 基于超图建模多站点之间的高阶关系；
- 基于 KAN（Kolmogorov-Arnold Network）提升模型表达能力与可解释性；
- 以多个时空预测基线模型作为对照，验证超图-KAN模型的有效性；
- 对温度、湿度、风速、云量等关键气象要素进行多步预测，并输出可用于论文分析的结果。

项目整体由两部分组成：

- `hyper_kan/`：主模型实验目录，用于承载超图-KAN相关实现与实验；
- `baseline/`：基线模型实验目录，用于统一运行 STGCN、GWNET、GRU、LSTM、STGNN 等对照模型。

该项目的目标不是只完成单一预测任务，而是建立一个可复现实验框架，使不同模型能够在相同数据、相同评价标准下进行公平比较，并进一步支撑“可解释性气象预测”的毕设论证。

## 3. 研究任务与建模目标

### 3.1 任务类型

本项目属于典型的气象时空序列预测任务，输入为历史多站点观测序列，输出为未来若干时间步的目标气象变量预测值。  
从建模角度看，任务同时包含以下特点：

- 时间相关性：气象变量具有明显的时间连续性和周期性；
- 空间相关性：不同站点之间存在空间传播和相互影响；
- 多变量耦合：目标变量会受到其他上下文变量影响；
- 高阶关系：传统图结构只能表示二元关系，而超图可以描述更复杂的群体关联；
- 可解释性需求：毕设不仅追求预测精度，还要求对模型决策依据进行分析。

### 3.2 任务目标

项目的主要任务目标包括：

1. 对温度、湿度、风速、云量等气象要素进行未来多步预测；
2. 在相同数据集和评价指标下，对超图-KAN与多种基线模型进行性能对比；
3. 分析超图结构与 KAN 表达在时空建模中的作用；
4. 生成训练日志、实验配置、误差指标和预测结果，为论文撰写提供支撑材料。

### 3.3 输出目标

每次实验通常需要输出以下内容：

- 训练日志：记录训练过程、参数、损失变化、验证表现；
- 配置快照：记录当前实验使用的完整配置；
- 模型结果：包括测试误差、预测值、真实值或中间结果；
- 对比结果：不同模型、不同气象要素下的指标表格；
- 可解释性分析素材：用于展示特征作用、关系建模效果或模型机制解释。

## 4. 数据集说明

这一部分是项目的核心基础。你的毕设不是“先有模型，后找数据”，而是“围绕数据结构设计合理的时空预测任务与模型比较体系”。因此，数据集说明必须写清楚“数据是什么、如何组织、如何进入模型、最终产出什么”。

### 4.1 数据集总体组成

当前项目的数据集位于项目根目录下的 `datasets/` 文件夹中，与 `baseline/`、`hyper_kan/` 处于同一级目录。  
这一设计的目的，是让主模型和基线模型都能共享同一份数据，而不是各自维护独立数据副本。

数据集按“气象要素”划分子目录。你当前毕设关注的主要要素为：

- 温度；
- 湿度；
- 风速或风分量；
- 云量。

每个要素目录内部至少包含三个 `pkl` 文件：

```text
某个要素目录/
├── trn.pkl      # 训练集
├── val.pkl      # 验证集
└── test.pkl     # 测试集
```

其含义为：

- `trn.pkl`：供模型参数学习使用；
- `val.pkl`：供模型调参、早停和选择最优模型使用；
- `test.pkl`：供最终性能评估和论文结果汇报使用。

### 4.2 数据集的组织逻辑

项目采用“按任务要素拆分”的组织方式，而不是将所有变量混在一个文件中。  
这样做有三个直接好处：

1. 便于分别研究不同气象要素的可预测性差异；
2. 便于多卡并行，让不同 GPU 同时跑不同要素；
3. 便于论文中分别展示温度、湿度、风速、云量的实验结果。

例如：

- GPU 0 跑温度；
- GPU 1 跑湿度；
- GPU 2 跑风速；
- GPU 3 跑云量；
- GPU 4、5 可继续跑其他模型或重复实验。

这与服务器部署方式是匹配的，因为每个进程只需要处理一个要素和一个模型组合，实验之间互不干扰。

### 4.3 数据文件格式说明

数据文件使用 `pkl` 格式存储。  
选择 `pkl` 的主要原因是：

- 可以直接保存 Python 中的复杂数据结构；
- 便于预处理后快速加载；
- 比纯文本格式更适合训练阶段频繁读取；
- 适合存储多维张量、索引、掩码、站点信息等内容。

在当前项目中，底层数据加载器直接读取 `trn.pkl`、`val.pkl`、`test.pkl`，不依赖必须存在的 `config.json`。  
这意味着数据集目录中即便只有 `pkl` 文件，也可以正常组织训练流程。

### 4.4 数据样本的典型组成

从当前基线运行日志可以确认，数据加载阶段至少包含如下概念：

- 目标气象变量；
- 上下文特征；
- 上下文特征选择掩码；
- 训练、验证、测试划分；
- 多站点时空序列数据。

日志中出现的信息包括：

```text
Context mask: [True, True, True, True, True, True, True, False]
Selected context features: 7/8
```

这说明当前数据处理流程中，至少存在 8 个上下文特征位，其中 7 个被启用，1 个被屏蔽。  
从论文写作角度，这一信息很重要，它反映出：

- 模型输入并不只有单一目标变量；
- 数据中包含外部辅助特征或上下文变量；
- 可以通过掩码控制哪些特征进入模型；
- 项目具备做消融实验和可解释性分析的条件。

虽然不同 `pkl` 文件的内部字段名称需要以实际数据结构为准，但从任务设计上可以将其理解为：

```text
单个样本 ≈ {
  历史输入序列,
  未来预测目标,
  站点维度信息,
  时间维度信息,
  上下文特征,
  可选的归一化或辅助元数据
}
```

### 4.5 数据的时空属性

本项目的数据不是普通的一维时间序列，而是典型的多站点时空序列数据。  
因此，每个样本通常至少包含以下维度逻辑：

- 时间维：历史观测窗口和未来预测窗口；
- 空间维：多个站点或网格节点；
- 特征维：目标变量及其辅助变量。

这意味着模型需要同时回答两个问题：

1. 历史变化趋势如何影响未来预测；
2. 不同站点之间的关系如何影响当前站点的未来状态。

基线模型中的图神经网络类模型，如 STGCN、GWNET、STGNN，主要针对第二类问题做建模；  
而超图-KAN模型则是在此基础上进一步增强高阶关系表达与可解释性。

### 4.6 数据集划分的意义

训练集、验证集、测试集三部分的职责不同：

- 训练集用于学习参数；
- 验证集用于调整超参数和防止过拟合；
- 测试集用于最终汇报结果。

这一划分方式对毕设很重要，因为它保证了：

- 模型选择过程与最终评估过程相互独立；
- 论文中的指标更可信；
- 可以公平比较主模型与各个基线模型。

如果论文中要说明实验严谨性，这一点必须明确写出。

### 4.7 数据目录与运行时解析方式

项目运行时，数据目录由配置系统动态解析。  
当前约定是从 `baseline/` 向上一级找到项目根目录，再进入 `datasets/`：

```text
baseline/
└── ../datasets/
```

因此，基线程序不会再强依赖 `baseline/raw_data/` 这样的默认路径，而是直接读取项目根目录下的共享数据集。  
这样做的优势包括：

- 与服务器部署结构一致；
- 更适合多实验模块共享数据；
- 避免复制数据带来的冗余和路径错误；
- 更方便批量运行和多卡并发。

### 4.8 数据集在论文中的可描述方式

在论文中，可以将本项目的数据集概括为：

> 本文实验数据按气象要素划分为多个独立子数据集，每个子数据集包含训练集、验证集和测试集三个部分，采用 `pkl` 格式存储多站点时空序列样本。各数据集对应温度、湿度、风速、云量等关键气象变量，并包含一定数量的上下文辅助特征。实验中，各模型在相同的数据划分和评价指标下进行训练与测试，以保证结果对比的公平性。

## 5. 模型体系说明

### 5.1 主模型：超图-KAN

`hyper_kan/` 目录中的主模型是本毕设的核心创新部分。  
其设计思路可以概括为：

- 利用超图结构表达多站点之间高阶关系；
- 利用 KAN 提升非线性拟合与函数表达能力；
- 在保持预测性能的同时，增强模型的可解释性；
- 为气象预测中的复杂关联提供更灵活的建模方式。

在论文中，超图-KAN可以定位为：

- 项目的主要研究对象；
- 与图神经网络、循环神经网络类基线模型对比的创新模型；
- 支撑“可解释性气象预测”这一课题名称的核心模型。

### 5.2 基线模型

`baseline/` 中的基线模型用于提供标准对照。  
从你当前实验日志可见，项目已至少支持以下模型：

- `STGCN`
- `GWNET`
- `GRU`
- `LSTM`
- `STGNN`

这些模型覆盖了不同建模思路：

- `GRU`、`LSTM`：偏时间序列建模；
- `STGCN`、`GWNET`、`STGNN`：偏图时空建模；
- `hyper_kan`：偏高阶关系与可解释性增强建模。

这种基线设置对论文有利，因为它兼顾了：

- 传统序列模型；
- 图结构时空模型；
- 创新主模型。

### 5.3 基线模型运行方式

基线实验由 `baseline/run_baseline.py` 统一调度。  
每次实验通常由“模型名 + 气象要素名”共同确定，例如：

```text
STGCN on Wind
GWNET on Temperature
LSTM on Humidity
```

程序会对每个实验组合自动完成：

- 创建输出目录；
- 记录日志；
- 保存配置；
- 加载数据；
- 执行训练、验证和测试；
- 输出实验结果。

## 6. 模型参数说明

模型参数可以分为四类：全局参数、数据参数、训练参数、模型专属参数。

### 6.1 全局参数

全局参数用于控制实验整体行为，常见包括：

- `seed`：随机种子，保证实验可复现；
- `gpu`：是否启用 GPU；
- `gpu_id`：当前进程使用的逻辑 GPU 编号；
- `output_dir`：实验输出目录；
- `dataset_dir`：数据集根目录；
- `exp_id` 或时间戳：用于区分不同实验输出。

其中，当前服务器环境采用多卡并发策略：  
进程启动时先通过 `CUDA_VISIBLE_DEVICES` 隔离物理卡，因此对 PyTorch 而言，逻辑设备编号通常固定为 `0`。

### 6.2 数据参数

数据参数控制样本如何进入模型，常见包括：

- `dataset`：当前使用的数据集名称；
- `input_window`：历史输入时间长度；
- `output_window`：未来预测时间长度；
- `train_sample_ratio`：训练数据采样比例；
- `context_mask`：上下文特征是否启用；
- `batch_size`：批大小；
- `num_nodes`：站点数或节点数；
- `feature_dim`：输入特征维度；
- `output_dim`：输出目标维度。

这些参数决定了模型看到的数据范围和预测跨度，是实验公平比较的重要基础。

### 6.3 训练参数

训练参数控制优化过程，典型包括：

- `learning_rate`
- `max_epoch`
- `optimizer`
- `weight_decay`
- `dropout`
- `patience`
- `early_stop`
- `clip_grad_norm`

这些参数影响模型收敛速度、泛化能力和训练稳定性。

### 6.4 模型专属参数

不同模型还会包含各自特有参数，例如：

- `GRU/LSTM`：`hidden_size`、`num_layers`；
- `STGCN`：时空卷积通道数、卷积核大小、层数；
- `GWNET`：扩张卷积结构、自适应邻接矩阵、残差通道；
- `STGNN`：图传播层数、空间聚合参数；
- `Hypergraph-KAN`：超边结构参数、KAN基函数相关参数、解释性模块参数。

论文中不需要把每个参数都平铺罗列，而应突出：

- 所有模型使用统一的数据划分；
- 主模型与基线模型在相近条件下训练；
- 关键参数设置来自统一配置文件和模型默认参数。
